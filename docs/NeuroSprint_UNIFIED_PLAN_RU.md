# NeuroSprint Unified Plan (RU)

Обновлено: 2026-03-08  
Статус документа: канонический рабочий roadmap

## 1. Назначение документа
Это главный рабочий документ проекта.

Только он должен использоваться как основной источник решений по:
- текущему продуктовому курсу;
- очередности реализации;
- статусу официальных и alpha-модулей;
- встраиванию `Progress System`.

Если другие старые roadmap/spec документы противоречат этому файлу, приоритет у этого документа.

## 2. Текущее состояние проекта
### Официальные production-ready модули
- Таблица Шульте
- Sprint Math
- Reaction
- N-Back Lite

### Частично интегрированные / не в текущем фокусе
- Pattern Recognition

### Alpha-линия
- Memory Match
- Block Pattern Recall
- Spatial Memory

## 3. Неподвижные правила
1. Доводить по одному новому модулю за раз до состояния `route -> setup -> session -> save -> stats`.
2. Не включать alpha-модули в официальные `stats / recommendation / challenge`, пока нет полного контракта.
3. Не начинать широкий рефакторинг без прямого продуктового выигрыша.
4. После изменения фактического статуса проекта обновлять только:
   - `docs/NeuroSprint_UNIFIED_PLAN_RU.md`
   - `docs/NeuroSprint_Execution_Status_RU.md`
   - `docs/CHANGELOG_RU.md`
5. Все устаревшие планы и старые спеки считать reference/архивом, а не рабочими инструкциями.

## 4. Зафиксированное решение по новым memory-модулям
- `Memory Match` — следующий production-кандидат.
- `Block Pattern Recall` — остаётся alpha-прототипом.
- `Spatial Memory` — hold / redesign.

Причина:
- `Memory Match` проще, чище и безопаснее как ближайший официальный модуль.
- `Block Pattern Recall` и `Spatial Memory` пока слишком близки по механике и требуют отдельной продуктовой развязки.

## 5. Актуальная дорожная карта
### Phase A — Production-доведение Memory Match
Цель:
- довести `Memory Match` до первого официального уровня качества.

Объём:
- setup / onboarding;
- игровой цикл;
- save результата;
- минимальная официальная аналитика;
- UX-polish, достаточный для production-ready статуса.

Не делать в этой фазе:
- параллельное production-доведение `Block Pattern Recall`;
- расширение `Spatial Memory`;
- широкий рефакторинг общего движка статистики.

### Phase B — Stabilization после Memory Match
Цель:
- проверить, что новый официальный модуль не ломает основной пользовательский контур.

Объём:
- build/test стабилизация;
- очистка спорных UX-мест;
- синхронизация статуса и changelog.

### Phase C — Progress System / Phase 1
Цель:
- встроить `Daily Training` как первый слой новой системы прогресса.

Объём:
- дневная тренировочная цель;
- факт выполнения;
- короткий понятный next step;
- только official-модули.

### Phase D — Progress System / Phase 2
Цель:
- добавить долгосрочную прогрессию через уровни и достижения.

Объём:
- уровни;
- достижения;
- безопасная связка с уже существующими официальными модулями.

### Phase E — Progress System / Phase 3
Цель:
- перевести результаты в карту навыков и понятный анализ сильных/слабых сторон.

Объём:
- skill map;
- интерпретация сильных сторон;
- интерпретация слабых сторон;
- без подключения alpha-линии.

### Phase F — Progress System / Phase 4
Цель:
- добавить AI-рекомендации как персональный guidance-слой.

Объём:
- рекомендации по следующему тренировочному шагу;
- опора на историю Daily Training и skill map;
- объяснение "что делать дальше и почему".

## 6. Progress System
### Состав системы
- Daily Training
- система уровней
- система достижений
- адаптивная сложность
- карта навыков
- анализ сильных и слабых сторон
- AI-рекомендации

### Порядок внедрения
1. `Phase 1 — Daily Training`
2. `Phase 2 — Levels + Achievements`
3. `Phase 3 — Skill Map + Strength/Weakness Analysis`
4. `Phase 4 — AI Recommendations`

### Ограничения внедрения
- Не подключать alpha-модули в официальный progress-loop.
- Не превращать систему прогресса в перегруженный gamification-слой.
- Сначала стабилизировать данные и official-модули, потом добавлять AI-рекомендации.

## 7. Что не делать сейчас
- Не начинать параллельно 2-3 новых игровых модуля.
- Не переписывать `sessionRepository` без явной необходимости.
- Не расширять официальную статистику alpha-модулями.
- Не использовать архивные roadmap/spec документы как рабочий источник решений.

## 8. Следующий конкретный шаг
1. Закрыть production-доведение `Memory Match`.
2. После стабилизации открыть `Phase C — Progress System / Phase 1 (Daily Training)`.
