# NeuroSprint: Execution Checklist для Feedback & Ideas

Назначение: этот документ нужен другому ИИ-агенту или инженеру, чтобы довести систему обратной связи и доски идей до рабочего состояния без поломки проекта.

Использовать вместе с:
- `docs/USER_FEEDBACK_AND_IDEAS_PLAN_RU.md`
- `docs/ACCOUNTS_AND_PROFILES_FINISHING_PLAN_RU.md`

Если между документами есть противоречие:
- продуктовая логика и scope — в `USER_FEEDBACK_AND_IDEAS_PLAN_RU.md`
- правила по аккаунтам и профилям — в `ACCOUNTS_AND_PROFILES_FINISHING_PLAN_RU.md`
- этот документ определяет только безопасный порядок исполнения.

## 1. Что уже есть и что нельзя перепридумывать

Перед началом считать зафиксированным:
- сайт уже работает на `Vercel`;
- аккаунты и профили уже подключены через `Supabase`;
- `Memory Match` уже имеет локальный `trainer feedback` слой;
- `siteAnalytics` уже существует и используется;
- `AppShell`, `Profiles`, `AuthContext`, `Supabase` wiring не нужно переписывать.

Не делать:
- не переписывать auth-систему;
- не убирать guest-режим;
- не переносить feedback в отдельный backend, если можно решить через `Vercel Functions`;
- не трогать `classes`, `competitions`, `group stats`, если это не регрессия;
- не менять маршруты аккаунтов и профилей;
- не делать форум, чат, комментарии к идеям или публичные рейтинги тренажёров;
- не использовать `SUPABASE_SERVICE_ROLE_KEY` в клиентском коде;
- не открывать guest direct insert policy в Supabase для feedback.

## 2. Правило маленьких шагов

Разработка должна идти фазами. После каждой фазы:
- запускать `npm run lint`
- запускать `npm run build`
- проверять, что сайт логически не сломан
- делать отдельный коммит

Не объединять несколько больших фаз в один коммит.

Рекомендуемый размер шага:
- одна серверная фаза,
- или один UI-слой,
- или один пилотный тренажёр.

## 3. Жёсткий порядок реализации

Следовать только в таком порядке:

1. Подготовка схемы данных
- добавить новый SQL-файл для feedback/ideas;
- создать таблицы `feedback_entries`, `idea_posts`, `idea_votes`;
- добавить индексы и ограничения;
- не менять существующие account/profile таблицы без необходимости.

2. Подготовка серверного слоя
- добавить `Vercel Functions` для `/api/feedback` и `/api/ideas*`;
- обновить `vercel.json`, чтобы `/api/*` не ломались из-за SPA rewrite;
- использовать `SUPABASE_SERVICE_ROLE_KEY` только на сервере.

3. Общий клиентский слой
- добавить client services для feedback и ideas;
- добавить helper для guest token;
- не вставлять прямые fetch-запросы по страницам хаотично.

4. Глобальная форма `Отзыв`
- встроить entry point в `AppShell`;
- реализовать modal/sheet;
- дать категории, текст, optional email;
- только после этого переходить к post-session feedback.

5. Post-session feedback pilot
- сначала `Memory Match`;
- потом `Sprint Math`;
- не подключать все тренажёры сразу.

6. Доска идей
- страница `/ideas`;
- public read;
- signed-in submit/vote;
- `Мои идеи` со статусом `На проверке`.

7. Аналитика
- добавить новые analytics events;
- не ломать существующие события.

8. Product-pass
- тексты;
- empty states;
- moderation states;
- auth prompts.

## 4. Обязательные продуктовые правила

Всегда соблюдать:
- `post-session feedback` = `звёзды 1–10 + причины + optional comment`
- звёзды не являются единственным видом feedback;
- глобальный `Отзыв` доступен и гостям, и пользователям с аккаунтом;
- доска идей доступна на чтение всем, но на запись только после входа;
- идеи проходят премодерацию;
- отзывы не публикуются наружу;
- публичный средний рейтинг тренажёров в v1 не показывать.

Пилотные модули строго:
- `Memory Match`
- `Sprint Math`

Если агент хочет подключить третий модуль в той же задаче:
- это уже выход за scope;
- нужно остановиться после второго пилота.

## 5. Антиспам и безопасность

Обязательно реализовать:
- honeypot для guest feedback;
- trimming и max length limits;
- rate limit по guest token;
- duplicate check для идей;
- duplicate/day limit для post-session feedback.

Нельзя:
- хранить сырой guest identifier как публичный ключ пользователя;
- доверять `account_id` из клиента без серверной проверки access token;
- показывать email автора на публичной доске идей.

Если реализация требует небезопасного client-side обхода:
- остановиться;
- не импровизировать;
- не внедрять workaround с секретом в браузере.

## 6. Stop conditions

Агент обязан остановиться и не продолжать “на удачу”, если:
- для реализации нужен client-side доступ к `service role`;
- `/api/*` не заработали после изменения `vercel.json`;
- ломается текущий auth/profile flow;
- ломается сохранение сессий тренажёров;
- не проходят `lint` или `build`;
- появляется необходимость трогать unrelated части проекта ради обхода проблемы.

В этих случаях агент должен:
- зафиксировать, на каком шаге остановился;
- перечислить конкретную ошибку;
- не продолжать расширение scope.

## 7. Что проверять после каждой фазы

После серверных изменений:
- `npm run build`
- проверить, что сборка Vite не сломана
- проверить, что `api`-маршруты не конфликтуют с SPA

После UI-изменений:
- открыть сайт локально
- убедиться, что `AppShell`, `Training`, `Profiles`, `Settings` не сломаны

После feedback-пилота:
- пройти тренировку в `Memory Match`
- убедиться, что feedback показывается корректно
- убедиться, что повторно в тот же день не показывается

После ideas board:
- гость видит идеи, но не может голосовать
- вошедший пользователь может создать идею
- новая идея не появляется публично сразу

## 8. Definition of Done

Задача считается завершённой только если выполнено всё:
- есть `feedback_entries`, `idea_posts`, `idea_votes` в облачной схеме;
- есть `POST /api/feedback`;
- есть `GET/POST /api/ideas` и vote endpoints;
- есть глобальная кнопка `Отзыв`;
- есть post-session feedback в `Memory Match`;
- есть post-session feedback в `Sprint Math`;
- есть страница `/ideas`;
- идеи требуют входа для создания и голосования;
- идеи публикуются только после модерации;
- есть analytics events;
- проходят `npm run lint` и `npm run build`;
- нет регрессий в auth/profiles и базовом training flow.

## 9. Формат работы другого агента

Рекомендуемый шаблон поведения:

1. Перед началом коротко перечислить текущую фазу.
2. Сделать только изменения этой фазы.
3. Прогнать проверки.
4. Коротко описать:
- что сделано,
- что не сделано,
- что осталось следующим шагом.

Если агент начинает:
- переписывать архитектуру,
- менять unrelated pages,
- трогать account/profile flows без необходимости,
- расширять scope до форума/чата,

значит он ушёл от плана и должен быть остановлен.
