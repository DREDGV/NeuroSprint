# NeuroSprint: Finishing Plan для аккаунтов и профилей

## Кратко
- Этот документ заменяет роль рабочего roadmap по теме `аккаунты + профили`.
- Текущая архитектура уже выбрана и не должна переписываться: `Supabase Auth + Supabase Postgres` для облачного слоя, `Dexie/IndexedDB` для guest-режима, локального кэша и офлайн-очереди.
- Цель документа: дать даже слабому агенту безопасный пошаговый план, по которому можно довести систему регистрации и профилей до public-ready состояния, не сломав существующий проект.

## Текущее состояние
- В проекте уже есть `AuthContext` и маршруты:
  - `/auth/login`
  - `/auth/register`
  - `/auth/forgot-password`
- `/profiles` уже работает как объединённый хаб аккаунта и тренировочных профилей.
- Локальные профили уже умеют быть `guest` и `linked`.
- Импорт локальных guest-профилей в аккаунт уже частично реализован.
- Logout уже блокирует linked-профили до следующего входа.
- Production Supabase-проект и Vercel env уже созданы и подключены.

## Главные правила для любого агента
- Не переписывать auth на самодельный JWT-backend.
- Не убирать guest-режим.
- Не смешивать `Account` и `TrainingProfile` в одну сущность.
- Не переносить управление аккаунтом в отдельный dashboard: основной account hub остаётся на `/profiles`.
- Не дублировать account forms в `/settings`: `Settings` — это device/app settings, а не account center.
- Не показывать и не разрешать self-service роли `teacher` и `admin`.
- Не класть `service_role` или другие секреты в клиентский код.
- Не трогать `classes`, `competitions`, `group stats`, feature flags и тренажёры, если это не требуется для исправления прямой регрессии от auth/profile.

## Source of truth
- `Supabase Auth` хранит:
  - учётную запись пользователя;
  - email;
  - auth session;
  - user metadata вроде `display_name`.
- `Supabase Postgres` хранит:
  - `accounts`
  - `training_profiles`
  - все cloud-таблицы прогресса и синхронизации из `docs/SUPABASE_ACCOUNTS_SCHEMA.sql`
- `Dexie/IndexedDB` хранит:
  - guest-профили;
  - локальный кэш linked-профилей;
  - pending-изменения до синхронизации.

## Целевая модель
- `Account` — это облачный аккаунт пользователя сайта.
- `TrainingProfile` — это профиль внутри NeuroSprint, с которым пользователь тренируется.
- Один `Account` может владеть несколькими `TrainingProfile`.
- Guest-профиль существует только локально, пока пользователь не привяжет его к аккаунту.
- Linked-профиль принадлежит аккаунту и синхронизируется через облако.

## Что должно считаться готовым v1
- Пользователь может:
  - зарегистрироваться;
  - войти;
  - выйти;
  - восстановить пароль;
  - увидеть понятный account/profile hub;
  - импортировать локальные профили;
  - продолжить работу с linked-профилями на другом устройстве.
- При logout:
  - guest-профили остаются доступны;
  - linked-профили видны, но заблокированы до входа.
- Sync не создаёт дублей при повторном импорте или повторной синхронизации.
- В UI нет технического жаргона вроде `linked`, `sync`, `Supabase`, `RLS`.

## Порядок работ

### Этап 0. Сделать этот документ единственной точкой истины
- Считать этот файл основным roadmap по теме.
- Старые документы:
  - `docs/ACCOUNTS_AND_PROFILES_PLAN_RU.md`
  - `docs/ACCOUNTS_PRODUCTION_ROLLOUT_RU.md`
  - `docs/SUPABASE_SETUP_GUIDE_RU.md`
  использовать как исторические справки, а не как рабочий план реализации.
- Если агент находит противоречие между этим документом и старым roadmap, побеждает этот документ.

### Этап 1. Довести account hub на `/profiles`
- Сделать `/profiles` единственной публичной точкой для:
  - `Создать аккаунт`
  - `Войти`
  - `Выйти`
  - `Импортировать профили`
  - `Выбрать активный профиль`
  - `Управлять аккаунтом`
- В верхней части страницы должны быть чётко разделены:
  - блок `Аккаунт`
  - блок `Тренировочные профили`
- В account block обязательно показывать:
  - email аккаунта;
  - display name;
  - статус аккаунта;
  - статус синхронизации;
  - следующий рекомендуемый шаг.
- Убрать из пользовательских текстов технические слова и объяснять всё через пользовательские сценарии:
  - `локально на этом устройстве`
  - `сохранено в аккаунте`
  - `нужен вход`
  - `можно продолжить позже`

### Этап 2. Довести import wizard и profile states
- Import локальных guest-профилей должен выглядеть как явный сценарий, а не как просто banner.
- В wizard должны быть состояния:
  - найдено локальных профилей;
  - выбранные профили;
  - импорт идёт;
  - импорт завершён;
  - импорт не удался, доступен retry.
- Импорт должен быть idempotent:
  - повторный запуск не создаёт дубли training profiles;
  - повторный запуск не создаёт дубли history/progress tables.
- После import список профилей и их статусы должны обновляться автоматически без ручного refresh.
- Locked linked-профили после logout должны:
  - оставаться видимыми;
  - иметь понятную подпись, почему они недоступны;
  - давать понятный CTA на вход.

### Этап 3. Довести account management
- Управление аккаунтом должно жить в `/profiles`, не в `/settings`.
- В `AuthContextValue` должен быть явный action для обновления account data:
  - `updateAccountProfile(payload: { displayName?: string; email?: string })`
- Пользователь должен уметь:
  - изменить display name;
  - инициировать смену email;
  - запросить reset password;
  - выйти из аккаунта.
- Display name должен обновляться:
  - в Supabase Auth metadata;
  - в `public.accounts`.
- После успешного обновления account state должен обновляться локально без hard reload.
- Если email change ждёт подтверждения по почте, UI должен показывать отдельное состояние `проверьте почту`.

### Этап 4. Довести sync и second-device flow
- Cloud остаётся source of truth для linked-профилей.
- Dexie остаётся локальным слоем и fallback-режимом.
- После login на новом устройстве приложение должно:
  - сначала подтянуть linked-профили из облака;
  - затем предложить выбрать активный профиль;
  - и только потом вести дальше в тренировки.
- Для linked-профилей должны быть понятные статусы:
  - `локально`
  - `ожидает отправки`
  - `синхронизировано`
  - `ошибка`
- При `ошибка` должен быть явный retry action.
- Если sync временно недоступен, локальные данные не должны пропадать.

### Этап 5. Production hardening
- Проверить и довести production auth flow:
  - email provider включён;
  - redirect URLs соответствуют продакшену и localhost;
  - SMTP реально работает;
  - reset password проходит end-to-end через live domain.
- Убедиться, что на production:
  - env берутся из `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY`;
  - приложение не падает без полного refresh;
  - ошибки auth/sync показываются как user-friendly сообщения, а не raw Supabase errors.

### Этап 6. Что не является launch-blocker, но должно быть отдельно зафиксировано
- Self-service удаление аккаунта не делать через browser-only код.
- Пока нет серверной прослойки, удаление аккаунта считать manual/admin-only операцией через Supabase dashboard.
- Если позже понадобится self-service account deletion:
  - делать только через server-side path;
  - preferred: Supabase Edge Function с service role secret;
  - fallback: существующий `backend/`, если он станет реально развёрнутым production service.
- Не пытаться использовать `auth.admin.deleteUser` из клиентского приложения.

## Интерфейсы и инварианты
- `AccountProfile` — только облачный аккаунт, не training profile.
- `User`/training profile должен продолжать содержать:
  - `accountId`
  - `remoteId`
  - `ownershipKind`
  - `syncState`
  - `lastSyncedAt`
  - `avatarEmoji`
- Вся логика import/sync должна оставаться централизованной вокруг account sync слоя, а не дублироваться по страницам.

## Что тестировать после каждого крупного шага

### Guest flow
- Создать локальный профиль.
- Пройти тренировку.
- Обновить страницу.
- Убедиться, что guest-профиль и локальные данные не потерялись.

### Register flow
- Зарегистрироваться на live site.
- Вернуться в `/profiles`.
- Увидеть account state и import prompt.

### Import flow
- Импортировать один guest-профиль.
- Импортировать несколько guest-профилей.
- Повторить import и убедиться, что дублей нет.

### Logout / login flow
- Выйти из аккаунта.
- Проверить, что linked-профили locked.
- Проверить, что guest-профили по-прежнему доступны.
- Войти снова и убедиться, что linked-профили снова открываются.

### Second-device flow
- На чистом устройстве войти в тот же аккаунт.
- Убедиться, что linked-профили подтягиваются из облака.
- Выбрать активный профиль и продолжить тренировки.

### Account management flow
- Изменить display name.
- Инициировать смену email.
- Запросить reset-password.
- Проверить, что пользовательские статусы и тексты остаются понятными.

### Failure flow
- Смоделировать временную ошибку sync.
- Проверить, что локальные данные не теряются.
- Проверить, что есть retry и понятное сообщение.

### Regression flow
- Проверить, что `/training`, `/stats`, `/settings` и скрытые feature-flag разделы не сломались от auth/profile изменений.

## Явные stop conditions
- Регистрация, вход, выход и reset password работают в production.
- `/profiles` понятен пользователю без технических терминов.
- Import wizard не создаёт дублей.
- Linked-профили корректно lock/unlock при logout/login.
- Sync на другом устройстве работает предсказуемо.
- SMTP и redirect flow проверены живыми тестами.
- В проекте нет client-side логики, которая пытается удалять auth users через секретные ключи.

## Assumptions
- Этот roadmap рассчитан на доведение текущей системы до public-ready состояния, а не на полную перестройку продукта.
- Для v1 launch-ready self-service account deletion не является обязательным блокером релиза.
- Managed Supabase остаётся production stack для auth и постоянной БД пользователей.
- Любой агент, продолжающий работу по этой теме, обязан сначала прочитать этот документ целиком, а уже потом вносить изменения в auth/profile слой.
