# Аккаунты NeuroSprint: production rollout и постоянное хранение данных

## Главное
- Аккаунты пользователей не должны храниться внутри фронтенда, `localStorage`, `Dexie` или Vercel deployment.
- Для production-режима NeuroSprint постоянная база пользователей должна жить во внешнем облачном сервисе:
  - `Supabase Auth` для email/паролей и сессий,
  - `Supabase Postgres` для профилей, прогресса и синхронизации.
- `Vercel` в этой схеме отвечает только за публикацию сайта. Обновления сайта не удаляют пользователей и не сбрасывают базу данных.

## Где будут храниться данные

### 1. Постоянные данные аккаунта
Хранятся в `Supabase`:
- учётная запись пользователя;
- email и данные входа;
- профили аккаунта;
- история тренировок, если профиль уже привязан к аккаунту;
- уровни, достижения, дневной прогресс и другие cloud-таблицы из схемы.

Это постоянное хранилище. Оно не теряется после:
- `git push`;
- нового деплоя в `Vercel`;
- обновления фронтенда;
- перезапуска сайта.

### 2. Локальные данные на устройстве
Хранятся в `Dexie / IndexedDB` внутри браузера:
- гостевые профили;
- локальный кэш привязанных профилей;
- очередь офлайн-изменений до синхронизации.

Это нужно для:
- гостевого режима без регистрации;
- офлайн-работы;
- плавной загрузки уже известных данных.

Но локальное хранилище не заменяет постоянную облачную БД. Если пользователь очистит данные браузера, локальные гостевые профили исчезнут. Для этого и нужен аккаунт.

## Почему база не будет теряться после обновлений
- `Vercel` деплоит статический фронтенд и клиентский код.
- `Supabase` хранит данные отдельно как managed сервис.
- При деплое сайт просто начинает обращаться к той же самой внешней БД.
- Пока не удалён сам проект Supabase и не сломаны ключи доступа, пользователи и их данные сохраняются.

Именно поэтому production-схема должна быть такой:

```text
Пользователь -> Сайт на Vercel -> Supabase Auth + Supabase Postgres
                           \
                            -> Dexie/IndexedDB как локальный кэш и guest-режим
```

## Что нужно включить до публичного релиза

### Обязательно
1. Создать production-проект в `Supabase`.
2. Применить схему из [SUPABASE_ACCOUNTS_SCHEMA.sql](./SUPABASE_ACCOUNTS_SCHEMA.sql).
3. Добавить env-переменные:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
4. Добавить эти же переменные в `Vercel` для production environment.
5. Настроить redirect URL для восстановления пароля.
6. Подключить SMTP для писем восстановления и подтверждения.
7. Прогнать сценарии:
   - регистрация;
   - вход;
   - сброс пароля;
   - импорт локальных профилей;
   - выход;
   - вход с другого устройства.

### Желательно
1. Включить backups или хотя бы экспорт схемы/данных по расписанию.
2. Проверить `RLS` на всех cloud-таблицах.
3. Создать отдельный production project, а не использовать dev-базу.
4. Завести отдельный email-адрес для системных писем.

## Что будет происходить с пользователем

### Без аккаунта
- пользователь создаёт локальный профиль;
- тренируется на одном устройстве;
- прогресс живёт только в браузере.

### После регистрации
- создаётся аккаунт в Supabase Auth;
- локальные профили можно импортировать в аккаунт;
- профиль становится доступен на других устройствах;
- изменения сначала сохраняются локально, потом синхронизируются в облако.

### После выхода
- гостевые профили остаются на устройстве;
- профили аккаунта становятся заблокированными до следующего входа;
- после входа они снова открываются и синхронизируются.

## Риски, которые нельзя игнорировать
- Если оставить только Dexie без облачной БД, пользователи будут терять данные после очистки браузера или смены устройства.
- Если хранить что-то критичное только в Vercel build artifacts, данные будут теряться при каждом деплое.
- Если не настроить SMTP, восстановление пароля будет формально существовать в коде, но не будет работать для реальных людей.
- Если не проверить `RLS`, есть риск неправильного доступа к чужим данным.

## Вывод
Для production NeuroSprint постоянная и стабильная база пользователей должна жить во внешнем `Supabase Postgres`, а не внутри фронтенда и не внутри деплой-платформы. `Dexie` остаётся локальным слоем удобства, а не главным источником истины.
