# NeuroSprint: понятный workflow разработки и деплоя

## Главная идея

У проекта должно быть три разных контура:

1. `local` — ваша разработка на компьютере
2. `preview` — тестовая онлайн-сборка по отдельной ветке
3. `production` — основной сайт для пользователей

Это позволяет:

- разрабатывать новые функции, не показывая их на основном сайте
- тестировать изменения в реальном интернете до публикации
- держать понятный процесс и для разработчика, и для будущих помощников

## Как устроить ветки

- `main` — только стабильная версия, которая идёт на основной домен
- `feature/...` — отдельные новые функции
- при желании позже можно добавить `staging` как постоянную тестовую ветку

Рекомендуемый сценарий:

1. Делаете новую функцию в ветке `feature/...`
2. Проверяете локально
3. Пушите ветку в GitHub
4. Vercel создаёт preview deployment
5. Проверяете функцию по preview URL
6. Когда всё готово — merge в `main`
7. После merge обновляется основной сайт

## Как скрытые функции не показывать пользователям

Для этого в проекте есть feature flags.

Сейчас подготовлены флаги:

- `classes_ui`
- `competitions_ui`
- `group_stats_ui`
- `online_competitions`

По умолчанию в production они должны быть выключены.

Это значит:

- код можно писать уже сейчас
- маршруты и кнопки можно тестировать локально
- на основном домене пользователи их не увидят, пока флаг не включён

## Где управлять скрытыми функциями

Есть два уровня управления.

### 1. Через env на deployment

Это основной уровень для продакшена и preview.

Пример:

```env
VITE_FEATURE_CLASSES_UI=false
VITE_FEATURE_COMPETITIONS_UI=false
VITE_FEATURE_GROUP_STATS_UI=false
VITE_FEATURE_ONLINE_COMPETITIONS=false
```

### 2. Через local overrides в браузере

Это режим для разработки и локального теста.

В приложении:

1. Откройте `Настройки`
2. Включите `Режим разработчика`
3. Используйте блок `Предпросмотр скрытых функций`

Эти переключатели:

- работают только в текущем браузере
- не влияют на пользователей
- удобны для локальной проверки ещё не опубликованных экранов

## Как тестировать новые функции в реальном поле

Есть три уровня тестирования.

### Уровень 1. Локально

Подходит для быстрой разработки:

```powershell
npm run dev
```

Потом в `Настройки` включаете нужные feature flags и проверяете UI.

### Уровень 2. Preview deployment

Подходит для реальной проверки в интернете:

1. Пушите ветку `feature/...`
2. Vercel создаёт preview URL
3. Вы открываете его с телефона, другого ноутбука или даёте нескольким тестерам

Если нужно, для preview можно включить флаги, которые в production выключены.

### Уровень 3. Production

Когда функция готова:

1. merge в `main`
2. убедиться, что нужный feature flag включён только если функция действительно готова
3. production обновляется

## Как развивать сетевые режимы позже

Для сетевых функций правильный путь такой:

1. сначала делаем UI и flow скрытыми под feature flags
2. потом подключаем backend на VPS
3. тестируем в preview deployment
4. после этого включаем флаги на production

То есть сетевые функции не должны сразу появляться на основном домене.

## Практическое правило на будущее

Любая новая большая функция проходит один и тот же путь:

1. код в `feature/...`
2. скрытие через feature flag
3. локальная проверка
4. preview deployment
5. полевое тестирование
6. merge в `main`
7. включение флага в production, когда функция готова

Это и есть самый понятный и безопасный способ вести проект дальше.
