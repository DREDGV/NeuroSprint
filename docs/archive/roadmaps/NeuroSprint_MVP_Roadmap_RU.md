# NeuroSprint MVP Roadmap (RU)

Обновлено: 2026-03-07
Текущая версия кода: `0.5.0-dev.5`

## 1. Текущее состояние проекта
- Проект находится в фазе стабилизации и выравнивания пользовательской поверхности после серии крупных функциональных этапов `v0.5-v0.7`.
- Последние завершённые сессии:
  - честная маркировка alpha-тренажёров в `TrainingHub`;
  - очистка user-facing текстов и проблемных строк;
  - упрощение верхней части `/stats`, `/stats/individual`, `/stats/group`.
- Главная текущая задача больше не в добавлении ещё одного набора функций, а в управляемом переходе к следующему модулю без расползания проекта.

## 2. Каталог модулей и их реальный статус
| Модуль | Статус | Комментарий |
|---|---|---|
| Таблица Шульте | Production-ready | Основной модуль, route/setup/session/stats работают. |
| Sprint Math | Production-ready | Полный пользовательский цикл и статистика работают. |
| Reaction | Production-ready | Рабочий модуль с несколькими режимами и интеграцией в статистику. |

## 3. Что уже считается завершённым
### Платформа и ядро
- `M0-M6` закрыты.
- Профили, роли, классы, PWA, базовые тестовые контуры работают.
- Простая и расширенная статистика существуют и используются в продукте.

### Игровые модули
- Закрыты и считаются официальными:
  - `v0.2`: Шульте `Classic+`, `Timed+`, `Reverse`;
  - `v0.5`: Sprint Math;
  - `v0.5.P-v0.7.A`: Reaction;
  - `v0.7.B`: N-Back Lite;
  - `v0.7.C`: Decision Rush.
- Memory Grid уже присутствует в коде и пользовательских маршрутах как рабочий модуль.

### Последняя стабилизация
- `TrainingHub` честно отделяет экспериментальные тренажёры от официальных.
- Простая статистика теперь сначала показывает краткий итог, затем тренд, затем детали.
- Индивидуальная и групповая статистика явно говорят, что alpha-модули туда пока не входят.

## 4. Что сделано частично
- Статистика улучшена по верхнему пользовательскому сценарию, но не полностью разгружена архитектурно.
- Pattern Recognition работает не на том же уровне продуктовой зрелости, что официальные модули.
- Alpha-линейка новых memory-тренажёров добавлена в код как прототипы, но не как официальная часть продукта.

## 5. Что ещё не начато или не доведено
- Production-доведение `Memory Match` через официальные контракты `setup -> session -> save -> stats`.
- Полноценное разведение `Spatial Memory` и `Block Pattern Recall` по механике и продуктовой ценности.
- Отдельная синхронная доработка Pattern Recognition как официального аналитического модуля.

## 6. Зафиксированные продуктовые решения
1. Официальная статистика показывает только официально поддерживаемые модули.
2. Alpha-модули не должны выглядеть как fully integrated части `stats / recommendation / challenge`.
3. Следующий новый модуль доводим по одному за раз.
4. Решение по 3 новым тренажёрам:
   - `Memory Match` — следующий production-кандидат;
   - `Block Pattern Recall` — остаётся alpha;
   - `Spatial Memory` — hold / redesign.

## 7. Ближайший порядок работ
### Основной следующий спринтовый шаг
- Начать production-доведение `Memory Match` без параллельного развития двух других alpha-модулей.

### Порядок
1. Зафиксировать контракты и маршрутную модель `Memory Match` как официального модуля.
2. Добавить сохранение сессий и минимальную официальную аналитику для `Memory Match`.
3. Оставить `Block Pattern Recall` в alpha-зоне без ложных обещаний полной поддержки.
4. Не трогать `Spatial Memory`, кроме отдельной продуктовой переработки концепции.

## 8. Progress System
Новая система развития пользователя встраивается поверх уже существующих модулей и статистики как отдельный продуктовый слой, а не как замена текущих тренировок.

### Состав системы
- Daily Training
- система уровней
- система достижений
- адаптивная сложность
- карта навыков
- анализ сильных и слабых сторон
- AI-рекомендации

### Этапы внедрения
#### Phase 1 — Daily Training
- Ввести ежедневный тренировочный контур как главный регулярный сценарий пользователя.
- Связать его с официальными production-ready модулями без подключения alpha-тренажёров.
- Сформировать простую логику: цель на день, факт выполнения, короткий next step.

#### Phase 2 — Levels + Achievements
- Добавить уровни как долгосрочную прогрессию поверх регулярных тренировок.
- Добавить достижения за последовательность, регулярность, личные рекорды и освоение модулей.
- Не превращать систему в перегруженный gamification-слой: уровни и достижения должны усиливать мотивацию, а не шум.

#### Phase 3 — Skill Map + Strength/Weakness Analysis
- Ввести карту навыков как понятное представление сильных и слабых сторон пользователя.
- Связать официальные модули с конкретными skill-направлениями.
- Добавить простую интерпретацию: что развито лучше, что проседает, где нужен фокус.

#### Phase 4 — AI Recommendations
- Поверх skill map и истории тренировок добавить AI-рекомендации.
- Рекомендации должны отвечать на вопрос: что тренировать дальше и почему.
- AI-слой внедрять только после появления устойчивых данных из Daily Training, Levels и Skill Map.

## 9. Риски
- Основной риск не в нехватке функций, а в рассинхроне между кодом, UI и документацией.
- Есть риск конфликтов из-за параллельных правок другими агентами; крупные сессии должны оставаться строго локальными по области изменений.
- Alpha-линейка уже видна пользователю, поэтому её статус нужно и дальше удерживать честным.
- Для Progress System отдельный риск в том, что она может быть начата слишком рано до стабилизации базовых модулей и официальных данных.

## 10. Next Session Start
1. Начать отдельную реализационную сессию по `Memory Match` как первому новому production-модулю.
2. Не расширять эту сессию на `Block Pattern Recall` и `Spatial Memory`.
3. После стабилизации `Memory Match` начать `Phase 1 — Daily Training` как первый шаг `Progress System`.
