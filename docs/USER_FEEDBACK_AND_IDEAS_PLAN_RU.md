# NeuroSprint: План системы обратной связи и доски идей v1

## Кратко
- Делаем не форум и не чат, а продуктовый слой обратной связи из трёх частей:
  - `post-session feedback` после тренировки,
  - глобальная форма `Отзыв`,
  - публичная `доска идей` с голосованием.
- В первый публичный релиз входят:
  - отзывы и рейтинги,
  - доска идей,
  - премодерация только для идей,
  - отзывы от гостей и пользователей с аккаунтом,
  - чтение идей для всех, создание и голосование только для вошедших пользователей.
- `Звёздный рейтинг 1–10` добавляется как дополнительный быстрый сигнал в `post-session feedback`, но не заменяет причины и комментарий.
- Новая система не ломает текущую архитектуру:
  - `Supabase` хранит постоянные данные отзывов и идей,
  - `Dexie/localStorage` остаётся только локальным UX-слоем,
  - гостевые отзывы пишутся не напрямую из браузера в Supabase, а через `server-side API`.
- Пилотный запуск `post-session feedback`:
  - `Memory Match`,
  - `Sprint Math`.
- В v1 не делать:
  - форум,
  - чат,
  - комментарии к идеям,
  - создание идей без входа,
  - встроенную админку модерации,
  - публичный рейтинг тренажёров на основе звёзд.

## Ключевые изменения

### 1. Пользовательские поверхности
- Добавить глобальную точку входа `Отзыв` в `AppShell`, но не как основной пункт `main nav`.
- Добавить отдельную публичную страницу `/ideas`:
  - читать идеи могут все,
  - предлагать идеи и голосовать могут только вошедшие пользователи.
- В `main nav` страницу `Идеи` не добавлять; вести на неё из:
  - глобальной кнопки `Отзыв`,
  - `Help`,
  - пустых состояний,
  - `Профили и аккаунт`.
- `post-session feedback` встроить только в:
  - `Memory Match`,
  - `Sprint Math`.
- Использовать существующий `TrainerFeedbackCard` как основу UI, а не переписывать его с нуля.
- Глобальный `Отзыв` делать как единый `sheet/modal`, а не как тяжёлую отдельную страницу.

### 2. Модель данных
- Разделить два потока:
  - `feedback_entries` — приватные отзывы о продукте и тренажёрах,
  - `idea_posts` и `idea_votes` — публичная доска идей.
- В `feedback_entries` хранить:
  - `submitter_kind`: `guest | account`,
  - `account_id?`,
  - `training_profile_id?`,
  - `guest_token_hash?`,
  - `source_surface`: `post_session | global_form`,
  - `category`: `bug | ux | idea | question | praise`,
  - `module_id?`,
  - `mode_id?`,
  - `route`,
  - `sentiment?`,
  - `star_rating?` со значением `1..10`,
  - `reasons text[]`,
  - `comment`,
  - `contact_email?`,
  - `client_context jsonb`,
  - `review_status`: `new | reviewed | archived`.
- В `idea_posts` хранить:
  - `author_account_id`,
  - `author_profile_id?`,
  - `title`,
  - `body`,
  - `category`: `training | ux | progress | social | account | stats | other`,
  - `moderation_status`: `pending | approved | rejected`,
  - `roadmap_status`: `new | planned | in_progress | done | declined`,
  - `rejection_note?`,
  - timestamps.
- В `idea_votes`:
  - один аккаунт = один голос на одну идею,
  - уникальность `(idea_id, account_id)`,
  - `downvote` в v1 не делать.
- На публичной доске показывать только идеи со статусом `approved`.

### 3. Хранение и серверный путь
- Не открывать прямой `anon insert` в Supabase для гостевых отзывов.
- Все операции записи делать через `server-side API` на Vercel:
  - `POST /api/feedback`
  - `GET /api/ideas`
  - `POST /api/ideas`
  - `POST /api/ideas/:id/vote`
  - `DELETE /api/ideas/:id/vote`
- Использовать `Vercel Functions` в `api/`, а не `Supabase Edge Functions`.
- Обязательно обновить `vercel.json`, чтобы `/api/*` не переписывались в `/index.html`.
- `server-side API` использует:
  - `SUPABASE_SERVICE_ROLE_KEY` только на сервере,
  - `Authorization` token пользователя для signed-in маршрутов,
  - серверную проверку пользователя через Supabase перед записью.
- Добавить server-only переменные окружения:
  - `SUPABASE_SERVICE_ROLE_KEY`
  - `FEEDBACK_GUEST_HASH_SECRET`
- Для гостей:
  - хранить стабильный guest token в браузере,
  - на сервер отправлять только token,
  - в БД хранить только его hash.
- Ни при каких условиях не выносить `service role` и серверную логику в клиент.

### 4. UX-правила
- Глобальная форма `Отзыв`:
  - доступна и гостям, и пользователям с аккаунтом,
  - категории: `Баг`, `Неудобно`, `Идея`, `Вопрос`, `Похвала`,
  - `comment` обязателен,
  - `contact_email` опционален,
  - для вошедшего пользователя email подставляется, но остаётся редактируемым.
- `post-session feedback`:
  - показывать только после результата тренировки,
  - структура: `звёзды 1–10 + причины + optional comment`,
  - шкалу визуально оформить как 10 кликабельных звёзд или 10-step rating row,
  - одному гостю или профилю не показывать повторно чаще одного раза в день на один модуль.
- Звёздный рейтинг использовать только как дополнительный продуктовый сигнал:
  - не делать его обязательным единственным фидбэком,
  - не использовать как публичный рейтинг тренажёра в v1,
  - применять для внутренней аналитики качества модулей.
- Доска идей:
  - для гостей только чтение,
  - CTA: `Войти, чтобы предложить идею` и `Войти, чтобы голосовать`,
  - для вошедшего пользователя:
    - создание идеи,
    - голосование и снятие голоса,
    - просмотр своих идей и их статусов.
- Премодерация:
  - новые идеи не публикуются сразу,
  - до одобрения они видны только автору в блоке `Мои идеи` со статусом `На проверке`,
  - встроенную модераторскую панель в v1 не делать; модерация идёт вручную через Supabase dashboard.
- Не использовать в пользовательском интерфейсе слова `RLS`, `service role`, `Supabase`.
- В v1 не давать пользователю редактировать или удалять уже отправленные идеи и отзывы.

### 5. Антиспам и защита
- Для гостевых отзывов в API добавить:
  - `honeypot` field,
  - trimming текста,
  - ограничения по длине,
  - мягкую проверку дублей.
- Ограничения на дубли:
  - `post-session feedback`: не больше одной записи на `guest/account + module + local day`,
  - глобальный гостевой отзыв: не больше 3 отправок за 24 часа на один guest token,
  - одинаковый комментарий в коротком окне отклонять.
- Для идей:
  - создавать можно только из аккаунта,
  - одинаковый нормализованный `title` от того же аккаунта в статусе `pending/approved` не создавать повторно.
- На публичной доске не показывать email автора.
- В качестве автора показывать:
  - `display name` аккаунта,
  - fallback: `Пользователь NeuroSprint`.
- Роль профиля на публичной доске не показывать.

### 6. Аналитика
- Не ломать текущую `siteAnalytics`.
- Добавить события:
  - `feedback_opened`
  - `feedback_submitted`
  - `feedback_dismissed`
  - `post_session_feedback_submitted`
  - `ideas_viewed`
  - `idea_submitted`
  - `idea_vote_added`
  - `idea_vote_removed`
- Для feedback-событий передавать:
  - `surface`,
  - `category`,
  - `module_id?`,
  - `submitter_kind`,
  - `star_rating?`.
- Для idea-событий передавать:
  - `category`,
  - `moderation_status`,
  - `roadmap_status`, если применимо.

## Публичные интерфейсы и контракты
- Новый маршрут:
  - `/ideas`
- Новые API endpoints:
  - `POST /api/feedback`
  - `GET /api/ideas`
  - `POST /api/ideas`
  - `POST /api/ideas/:id/vote`
  - `DELETE /api/ideas/:id/vote`
- Новые клиентские типы:
  - `FeedbackCategory`
  - `FeedbackSurface`
  - `FeedbackSubmitPayload`
  - `IdeaCategory`
  - `IdeaModerationStatus`
  - `IdeaRoadmapStatus`
  - `IdeaSummary`
  - `IdeaVoteState`
- Существующий `TrainerFeedbackCard` остаётся, но его submit path должен перейти с локального хранения на новый сервис отправки feedback.
- Существующий `trainerFeedbackRepository` оставить только для локальных UX-хелперов:
  - хранение факта `показано/обработано`,
  - больше не использовать как постоянное хранилище отзывов.

## Тесты и сценарии
- Гость может отправить глобальный отзыв без аккаунта.
- Гость может отправить `post-session feedback` в `Memory Match`, выбрать рейтинг `1–10`, и повторный показ в тот же день не появляется.
- Вошедший пользователь может отправить глобальный отзыв и `post-session feedback`, и запись сохраняется с account context.
- Вошедший пользователь может открыть `/ideas`, создать идею и увидеть её в `Мои идеи` со статусом `На проверке`, но идея не появляется публично сразу.
- Гость может открыть `/ideas` и читать одобренные идеи, но не может голосовать или создавать без входа.
- Вошедший пользователь может поставить один голос и снять его; дубли голосов не создаются.
- Повторная идея с тем же названием от того же аккаунта отклоняется с понятным сообщением.
- API отклоняет `honeypot`, слишком длинные поля и спам-повторы от гостя.
- `/api/*` продолжает работать после деплоя на Vercel и не ломается из-за SPA rewrites.
- Текущий auth/profile flow и сохранение сессий в `Memory Match` и `Sprint Math` не получают регрессий.
- Существующая аналитика продолжает работать, а новые feedback/idea события появляются в Vercel Analytics.

## Допущения
- Это launch-oriented план; форум и чат в него не входят.
- Публичная доска идей доступна на чтение всем, а на запись только пользователям с аккаунтом.
- Отзывы разрешены и гостям, и пользователям с аккаунтом.
- Идеи проходят премодерацию; отзывы остаются приватным внутренним сигналом.
- Пилотные модули для `post-session feedback`:
  - `Memory Match`
  - `Sprint Math`
- Модерация в v1 ручная через Supabase dashboard; встроенной админки нет.
- Self-service удаление и редактирование отправленных идей и отзывов в v1 не входят.
